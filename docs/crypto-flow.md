# Cryptographic flow

## Algorithm

**Ed25519**, via Node's built-in `crypto` module. Chosen because:

- Small keys (32 bytes) and signatures (64 bytes) → short tokens users can copy-paste.
- Deterministic signatures → no RNG failure mode during signing.
- Supported natively in Node with no external dependency.

## Signed payload

The bytes that get signed are literally:

```
<nonce>|<dob>
```

where `<nonce>` is the random challenge from the consumer (UUID v4 by default) and `<dob>`
is an ISO-8601 date string `YYYY-MM-DD`.

Pipe-delimited text is used instead of JSON on purpose: it removes any question of key
ordering, whitespace, or unicode normalization. Both sides just concatenate the same two
strings with a `|` and sign/verify those bytes.

> To change the claim to "over 18 only" later, change the payload to `<nonce>|over18:true`
> and update verify accordingly.

## Token format

The provider returns a URL-safe base64 string decoding to:

```json
{
  "nonce": "5a3f…",
  "dob":   "1992-07-14",
  "kid":   "k_2026-04-22T10:15:00Z_a7f1",
  "sig":   "base64url-ed25519-signature"
}
```

`kid` identifies which of the provider's published public keys the consumer should use to
verify. The consumer rejects the token if `kid` is not in its current cache.

## Key rotation timeline

Let `T = KEY_TIMESPAN_MS` (default 60s).

```
time ──▶

  key A   [────── current ──────][────── retired ──────]
  key B                          [────── current ──────][── retired ──]
  key C                                                 [── current ──]

         ▲                       ▲                       ▲
         rotation 0              rotation 1              rotation 2
         (A current, B next)     (B current, C next)     (C current, D next)
```

At every rotation:
1. The old `current` is discarded.
2. `next` is promoted to `current`.
3. A fresh key pair is generated and becomes the new `next`.

## Consumer refresh strategy

Consumers fetch `/keys` on a **cron timer** at `T/2 ± 20%` jitter, owned by the broker
module and booted from the Next `instrumentation` hook at server start. Jitter prevents
the thundering herd that would appear if all consumers synchronized on `T/2` boundaries.

**Verify requests never trigger a fetch.** The broker is the only code path that calls
`/keys`; `verifyToken` just reads whatever is in the cache. This is deliberate: if a
verify request could cause a `/keys` fetch, a provider could time-correlate the fetch
with a nearby signing request and defeat the unlinkability property of the manual flow.
If the cache is somehow empty at verify time (only possible before the first cron tick
completes at boot), verify blocks on that first fetch — but still never initiates its
own.

Because the exact rotation moment is not advertised, there is a short window after a
rotation where a consumer may still hold only the *retired* key plus what was formerly the
`next` (now `current`). This is fine: the `next` key was pre-published precisely so that
signatures produced immediately after rotation are still verifiable.

## Nonce / replay protection

The nonce is generated by the consumer and is what makes a token single-use. The consumer
stores the outstanding nonce in React state and checks that the token's `nonce` matches
what it issued. Once a nonce is used it is discarded.

For the easy flow the nonce travels in the URL, so the callback URL must be treated as a
capability token for the duration of the flow.

## Security caveats (this is a demo)

- No TLS: everything is `http://localhost`.
- No provider authentication: the "login" dropdown is faked.
- No consumer registration: any party can fetch keys and verify; that's fine because this
  system is anonymous-consumer by design, but it also means there is no rate limiting.
- Private keys are regenerated on every server restart and never persisted — tokens issued
  before a restart become unverifiable afterwards. Good enough for a demo, not for prod.
